## 0. 執行守則（硬性，缺一不可）

1. **開工前**：`git fetch origin` + `git status` 確認工作樹乾淨且與 origin/main 同步（有新 push 先 `git reset --hard origin/main`）。逐條開工前先重讀該 finding 的行號段落，確認現況相符。

3. **每完成一個 commit 批次**：`python -m pytest -q` 全綠才進下一批（基準 1185 collected）。每個 fix 都要加本文件指定的回歸測試（各區已有現成 harness：`test_autoclock_clock_safety.py`、`test_consult_decision_logic.py`（含 `_poll_harness`）、`test_alert_email_dedup.py`、`test_abbrev_engine.py`、`test_watchdog_core.py`、`test_roster_*.py`）。

5. **高風險域紀律**：
   - **打卡（autoclock）**：錯打/漏打/重複打卡＝實際傷害。任何行為變更除單元測試外，需用 `dry_run`（測試模式）實機走一次流程；不可放寬任何「不確定就不打」的既有防線。

   - **會診/止掛通知**：語意慣例「寧多寄不漏寄」（fail-open）；修正不可引入新的靜默不寄路徑。
6. 語意慣例：唯讀路徑 fail-open、寫入/資料落地 fail-closed；Tk 元件只在主執行緒碰（worker 用 `widget.after`）；roster 的 day_locks 契約「使用者鎖了就絕不無聲改動/刪除」。
7. Windows 陷阱：檔案讀寫一律 `encoding='utf-8'`；console print 中文可能爆（.pyw 無 console），測試斷言不依賴 console 輸出；subprocess 對 git/外部指令比照 `gitsync_storage._git` 的 `encoding="utf-8", errors="replace"` 慣例。
8. 修 `src/` 後不需自己碰 version.py / manifest.json——push_helper 會處理。
9. 安全紅線：code repo 維持 **public**（診間自動更新依賴 raw.githubusercontent.com）；`settings/` 與任何含 PII（員編/GSM/姓名/帳密）的檔案絕不可 commit。**CQ-04 要把硬編碼帳密移出源碼，不是把它搬進另一個會被 commit 的地方。**

## 2. 逐項修法規格

### 批次 1：打卡安全網（AC-01, AC-02, AC-09）

#### AC-01 [P1] 打卡點擊前無「當下時間仍在窗內」防線
- **檔案**：`src/autoclock.py`（perform_clock_action 約 948-1030；process_clock_task 約 1067-1177）
- **觸發情境**：任務在窗尾起跑（07:59 tick）而 portal 緩慢——`login()` 內建 5 次重試（每次 30s page-load + 20s wait + 退避），`perform_clock_action` 外層再重試 5 次（退避 2→60s），單帳號最壞可拖 15 分鐘以上；或前一任務占住 `clock_lock`，下一窗 worker 排隊到窗外才執行。
- **錯誤行為**：登入成功後照樣 `delay=random.randint(1,5)` → click execute——**全程無任何 `datetime.now()` 對 `check_end` 的檢查**（已抽驗屬實：唯一時間 gate 是 `_scheduler_tick` 的 `get_sched_key()`，只管起跑不管執行中）。上班卡打在 08:03＝遲到紀錄；midday_out 卡住時 midday_in worker 13:05 拿到鎖照打「上班」。
- **修法**：(1) `process_clock_task` 帳號迴圈每帳號開始前檢查 `datetime.now().time() > check_end` 即 break（記 warning + 走既有補卡提醒）；(2) `perform_clock_action` 在點擊 execute 前加同樣檢查（留 30-60 秒緩衝），超窗不點擊、記 warning。**兩道都要**（帳號間與登入後各一道）。
- **驗收測試**：`tests/test_autoclock_clock_safety.py` 加：注入 fake clock 使「進入時在窗內、點擊前已超窗」→ 斷言不點擊且補卡提醒觸發；正常窗內路徑回歸不受影響。另 dry_run 實機走一次。

#### AC-02 [P2] 畸形帳號設定 → 每 tick 整批崩潰、全帳號漏打卡零通知
- **檔案**：`src/autoclock.py`（約 1099-1100、1048、497-519）
- **觸發情境**：手改 `autoclock_config.json` 把某帳號 `schedule` 寫成 `null`，或清單混入非物件項。`a.get("schedule", {})` 對「key 存在但值為 null」回 `None` → `.get` AttributeError；該行在 try 之外 → 每分鐘 tick 都崩 → 全帳號漏打卡。`_missed_clock_check` 同款 comprehension 也炸 → 連補卡提醒都不跳。`acc["username"]`（1048）對缺 username 項二次 KeyError。
- **修法**：`load_config()` 回傳前消毒：剔除非 dict 項、`schedule` 非 dict 以 `{}` 取代（保留 W13 警告）；1048 改 `acc.get("username", "?")`。
- **驗收測試**：config 塞 `{"username":"x","password":"y","schedule":null}` 與字串項 → tick 不崩、正常帳號照打、警告照出。

#### AC-09 [P3] 密碼錯誤未分類為不可重試
- **檔案**：`src/autoclock.py`（login 約 581-591）
- **觸發情境**：改密碼後忘記更新設定。`login()` 只給錯誤元素 2 秒且不處理 JS alert（`punch_status.py:180-184` 有處理、這裡沒有）→ 錯密碼走 TimeoutException 當網路錯誤重試 5×5 次，整窗反覆登入+toast 轟炸+portal 帳號鎖定風險。
- **修法**：`login()` 加 alert accept 分支、將帳密錯誤升級為獨立例外型別；`perform_clock_action` 遇該型別單次醒目通知、當窗不再重試（對齊 `punch_status._is_retryable_punch_error` 的既有分類邏輯）。
- **驗收測試**：mock 錯密碼路徑 → 只通知一次、log 明確「帳密錯誤」、無反覆重試。

#### CQ-02 [P2] signature 對整段信文掃 `\d{6,}` → 內文雜數字假新會診
- **檔案**：`src/consult_query.py`（`_CHART_RE` :371；`text = roster+body` 合併 :1583-1584）
- **觸發情境**：逐病人內文擷取有 25s 總逾時與「未穩定即停」→ 同批病人各輪擷取到的內文段數漂移；某輪首次擷取到含身分證/手機/日期（≥6 連續數字）的病情摘要 → signature 增生 → 誤判新會診 → 重複寄全清單。agent 已實測 `re.findall(r"\d{6,}", "身分證A123456789 電話0912345678 於20260615入院")` 撈出 3 個假病歷號。
- **修法**：signature 只算 roster——用 CQ-01 的 `roster_texts` 通道，`_consult_signature` 改吃 roster 列（逐列取 `_ROSTER_ROW_RE` 的 chart 欄）。
- **驗收測試**：兩輪 poll 同一病人，第一輪 body 空、第二輪 body 含 "0912345678" → 第二輪不得寄信。

#### CQ-03 [P2] 手動觸發+擷取失敗 → 空集合覆寫基準 → 下輪重複寄全清單

- **驗收測試**：基準 `{"1234567"}`、手動觸發 extracted_text="" 寄信成功 → 基準仍為 `{"1234567"}`。
