### RF-01 [P1] 【F1 確認】只在 __init__ pull 一次、push 前不 pull → 雙機長駐編輯必然分歧，且分歧後永久靜默失敗、UI 零提示

- **檔案**：`src/cmuh_common/roster/gitsync_storage.py`（35-56, 102-117）

**問題**：
觸發情境：A、B 兩台早上各自開啟程式（各 pull 一次），整天長駐。A 中午改班 → commit+push 成功；B 整天看的是早上的舊資料（service 無快取、UI refresh 會重讀盤，但盤上檔案只有 __init__ 那次 pull 會更新）。B 下午也改班 → 本地 commit 成功，但 push 因 non-fast-forward 被拒，只寫 logging.warning（gitsync_storage.py:114-117），B 的使用者完全不知道「存檔沒同步出去」。此後兩機歷史分歧：B 隔天重開，__init__ 的 `git pull --ff-only`（:49）因分歧永遠失敗（:53-56 也只寫 log），B 從此永久活在自己的分支上——兩台班表各自演化、互相看不到，實務上等於排錯班；且無任何 UI 徵兆，要翻 automation_ui.log 才會發現。

**讀碼證據**：
gitsync_storage.py:35-36 `if self._git_ok and self._remote_sync: self._pull()` 是唯一的 pull 呼叫點；_push()（:102-117）直接 `git push remote HEAD`，push 前無 fetch/pull；push 失敗與 pull 失敗都只有 logging.warning，storage 沒有任何狀態欄位或 callback 讓 scheduler.py 的 UI 得知。scheduler.py:110 建構時也沒傳任何通知 hook。

**修法規格**：
gitsync_storage.py：(1) 新增 `self._git_lock = threading.RLock()`（與 F2 共用）與同步狀態機制：建構子加 `on_sync_state=None, on_remote_change=None, pull_interval_sec=300` 三參數；新增 `self.sync_state="ok"` 屬性與 `_set_state(state, detail)`（state ∈ ok/offline/diverged/error；更新屬性、logging、callback 非 None 時呼叫——文件註明 callback 會在背景 thread 執行，UI 端自行 marshal）。(2) _push() 全程 `with self._git_lock`，且推前先同步：先 `git fetch <remote>`（returncode!=0 → _set_state("offline", stderr) 後 return，離線不硬推）；再 `git merge --ff-only FETCH_HEAD`；merge 失敗（=分歧）→ 嘗試 `git pull --rebase`（兩台改不同 JSON 檔時可全自動復原）；rebase 也失敗 → `git rebase --abort` + _set_state("diverged", ...) 後 return（同檔衝突交人工，絕不自動 merge JSON）；最後 `git push <remote> HEAD`，成功 → _set_state("ok")。(3) 週期性 pull：__init__ 在 git_ok 時啟動 daemon Thread，每 pull_interval_sec 秒 `with self._git_lock:` 跑 fetch+ff-only merge（比對前後 `git rev-parse HEAD`，有變 → 呼叫 on_remote_change）；加 `self._stop_evt = threading.Event()`，用 `self._stop_evt.wait(pull_interval_sec)` 當 sleep，flush() 開頭 `self._stop_evt.set()` 收掉。scheduler.py：(a) :110 改 `GitSyncStorage(..., on_sync_state=self._on_sync_state, on_remote_change=self._on_remote_change)`；(b) _build_ui 在版本號 Label 旁加 `self._sync_label`（ttk.Label，預設空字串）；(c) `_on_sync_state(state, detail)` 用 `self.root.after(0, ...)` 更新 label 文案（ok→"已同步"灰、offline→"離線,變更僅存本機"橘、diverged→"同步衝突!"紅），且 state=="diverged" 第一次（用 self._diverged_warned flag 防重複）跳 messagebox.showwarning，文案含復原指引：「本機與另一台電腦的排班各自有修改且自動合併失敗。請保留一台繼續編輯，另一台關閉程式後在 settings/roster 資料夾執行 git pull --rebase 並解決衝突，完成前該台的修改不會同步。」；(d) `_on_remote_change` → `self.root.after(0, self._on_settings_changed)`（既有方法會 refresh 所有已建 duty tab；service 無快取、load_month 每次讀盤，已驗證）。

**⚠️ 驗證者對修法的修正/補強（照此調整）**：

### RF-06 [P2] 【F2 確認】_push 未持鎖：去抖 timer 與 flush() 可同時跑兩個 git push（cancel 對已起跑的 timer 無效）

- **檔案**：`src/cmuh_common/roster/gitsync_storage.py`（102-127）

**問題**：
flush()（:119-127）在 with self._push_lock 內 cancel timer 後「鎖外」呼叫 _push()；若 timer 已經開跑（threading.Timer.cancel 對已 fire 的 timer 是 no-op），timer thread 的 _push 與 flush 的 _push 併發執行兩個 `git push` 子行程。結果通常是其一被 remote 拒絕（stale info）產生誤導性 warning log；搭配 F1 的 pull-before-push 修法後危害升級：兩個 thread 併發 fetch/merge/rebase 會互撞 .git 的 ref lock。另外 timer thread 的 _push 與 UI thread 的 _commit（git add -A + commit）也是無鎖併發，目前 git 自身容忍，但同樣在加入 pull-before-push 後不可接受（pull 會 checkout 檔案，與 add/commit 競態）。

**讀碼證據**：
_schedule_push（:94-100）只用 _push_lock 保護 timer 欄位；_push（:102-117）本體完全無鎖；flush（:123-127）`with self._push_lock:` 區塊只涵蓋 cancel，`self._push()` 在鎖外。

**修法規格**：
沿用 F1 的 `self._git_lock = threading.RLock()`：(1) _push() 全程 `with self._git_lock`（F1 已含）；(2) _pull()、週期 pull 同樣持鎖；(3) _save 的 commit 段改為 `if not self._git_lock.acquire(timeout=10): logging.warning(...); self._schedule_push(); return`（拿不到鎖=背景正在 push/pull，檔案已由 super()._save 寫盤，下次 commit 的 add -A 會補收，不丟資料、不卡 UI 超過 10 秒），拿到則 try/finally release 內做 _commit+_schedule_push。_push_lock 保留只管 timer 欄位即可。

**⚠️ 驗證者對修法的修正/補強（照此調整）**：
兩處補強：(1) _save 的 _git_lock.acquire(timeout=10) 改為 timeout=2~3 秒 — _git() 子行程 timeout 30 秒（gitsync_storage.py:45），背景 push 卡網路時 UI thread 凍 10 秒對診間長駐工具太久；拿不到鎖時檔案已寫盤、下次 add -A 補收，縮短無害。(2) flush() 必須在 blocking 取得 _git_lock 後先做一次 catch-up commit（add -A + commit，沿用 _commit 的 nothing-to-commit=通過邏輯）再 _push()——否則 _save 因鎖 timeout skip commit 後若沒有下一次存檔，該變更永遠只在本機磁碟、關閉程式也不會被 commit/push（現行 flush 只 push 不 commit，scheduler.py:237 是關閉前唯一補救點）。其餘規格（_push 全程持 _git_lock、_pull/週期 pull 持鎖、_push_lock 只管 timer 欄位、skip 時仍 _schedule_push）核對無誤可照做。

**驗收測試**：
monkeypatch st._git 使 push 分支 sleep 0.5s 並用 threading.Barrier/計數器記錄併發度；同時從兩個 thread 呼叫 flush() 與直接呼叫 _push()，assert 任一時刻最多一個 git 操作在跑（計數器峰值==1）。

<details><summary>對抗驗證紀錄</summary>

讀碼證據全部成立：(a) gitsync_storage.py:123-127 的 with self._push_lock 區塊只包 cancel 與置 None，self._push() 在 :127 鎖外；(b) threading.Timer.cancel 對已 fire 的 timer 是 no-op（CPython 語意）；(c) 併發情境真實：scheduler.py:234-237 mainloop 結束即呼叫 storage.flush()，使用者存檔後 3 秒內 timer 起跑 _push（網路 push 可持續數秒）再關閉視窗，即兩個 git push 併發；_schedule_push（:94-100）在舊 timer 已 fire 時 cancel 無效又排新 timer，同樣可重疊；(d) _push 本體（:102-117）完全無鎖，與 UI thread 的 _commit（add -A + commit）無鎖併發屬實。嚴重度校正：單獨看，兩個併發 push 推同一 HEAD，git remote ref lock 會讓其一拿到 stale-info 拒絕或 Everything up-to-date，僅誤導性 warning、不丟資料——單獨評 P3；但此鎖是 F1（pull-before-push）修法的前置必要條件：無鎖下兩個併發 pull 搶 .git/index.lock、且 pull 改 working tree 期間 UI thread add -A 可能把合併半途狀態 commit 並推上 remote，屬真實資料風險，兩者必須同批修，故 P2 維持。

### RF-13 [P2] pythonw(.pyw)模式下每個 git 子行程都會閃出黑色 console 視窗:開檔 1 次(pull)、每次存檔 2 次(add+commit,且在 UI 執行緒)、每次去抖推送 2 次(remote+push)

**問題**：

**讀碼證據**：

**修法規格**：

```python
def _git(self, *args, timeout: float = 30.0) -> subprocess.CompletedProcess:

        kwargs["creationflags"] = 0x08000000  # CREATE_NO_WINDOW:pythonw 下不閃 console

        ["git", "-C", self.base_dir, *args],
        capture_output=True, text=True, timeout=timeout, check=False, **kwargs)
```

```python
def _git(self, *args, timeout: float = 30.0) -> subprocess.CompletedProcess:

        capture_output=True, text=True, timeout=timeout, check=False,
        creationflags=getattr(subprocess, "CREATE_NO_WINDOW", 0))
```

### RF-14 [P2] sync_members 作廢餘額但不清 history 內該員 deltas，移除→重加→同月 resettle 會憑空生出欠款（帳本損壞，下月目標被墊高≈多排班）

**讀碼證據**：

purge 迴圈必須過濾 scope，否則同 id 存在於另一 scope 時會造成鏡像損壞（rollback 找不到 delta → resettle 雙重計入、正向憑空生點）。正確寫法（放在 sync_members 的 `del book[mid]` 之後）：`for e in (ledger.get("history") or []):` → `if e.get("scope") == scope:` → `(e.get("deltas") or {}).pop(mid, None)`。其餘照原提案：只改 sync_members，reset_member 勿動。建議補一個 pytest 案例：settle→sync 移除→sync 加回→同月 settle，斷言被移除重加者餘額為 0 而非負值；再補一個跨 scope 案例斷言另一 scope 的 history deltas 未被清。

**P3-01** `src/cmuh_common/roster/gitsync_storage.py`:42-45 — _git 的 text=True 未指定 encoding：Windows 以 cp950 strict 解碼 git 輸出，UnicodeDecodeError(ValueError) 不在 except 清單內，會炸到 UI thread 的 _save

subprocess.run(text=True) 在此機以 locale 編碼（cp950）strict 解碼 stdout/stderr；git 輸出若含非 cp950 位元組（如 hint 訊息、非 ASCII 路徑、或使用者 repo 路徑含特殊字），communicate 內拋 UnicodeDecodeError——它是 ValueError 子類，不被 `except (OSError, subprocess.SubprocessError)` 接住，會從 _commit 一路傳到 UI thread 的存檔呼叫端（super()._save 已完成，資料不丟，但 UI 收到莫名例外）。此外 :79 用英文字串 "nothing to commit" 判斷乾淨樹，git 若被 locale 本地化（MSYS2 中文 locale）會誤判成真失敗 → 永遠不排 push。本專案 memory 已有 cp950 前科，值得一次修掉。

修法方向：_git() 改為：`env = {**os.environ, "GIT_TERMINAL_PROMPT": "0", "LC_ALL": "C"}`，subprocess.run 加 `encoding="utf-8", errors="replace", env=env`（移除 text=True，encoding 已隱含 text mode）。GIT_TERMINAL_PROMPT=0 同時修掉另一風險：GUI 程式無 console 時 git 等認證輸入會卡滿 30s timeout（__init__ 的 pull 在主 thread → 啟動白畫面 30 秒），設 0 讓它立即失敗走 offline 路徑。LC_ALL=C 保證 :79 的英文訊息解析穩定。


**P3-02** `src/cmuh_common/roster/gitsync_storage.py`:39-40 — _is_git_repo 只認 .git「目錄」：git worktree / submodule 的 .git 是檔案 → 靜默退化成純本機 storage，使用者以為有同步

使用者若用 `git worktree add` 或把 roster 目錄設成 submodule，base_dir/.git 是一個指向 gitdir 的「檔案」，os.path.isdir 回 False → _git_ok=False → 所有 git 動作靜默跳過，且無任何 log（退化路徑連 info 都沒印），使用者以為兩台在同步，實際各存各的。

修法方向：改成兩段：`p = os.path.join(self.base_dir, ".git")`，`ok = os.path.exists(p)`（檔案或目錄皆可；worktree/submodule 的 .git 檔案存在即真，git -C 自會解析）；並在 __init__ 加一行 `logging.info("[roster.gitsync] git 同步: %s", "啟用" if self._git_ok else "未啟用（%s 非 git repo）" % self.base_dir)`，讓退化路徑至少可從 log 確認。


**P3-03** `src/scheduler.py`:234-239 — 【F4 確認・可接受】flush 只在 mainloop 正常返回時執行；硬殺（watchdog/工作管理員/斷電）漏推的 commit 要等「下一次存檔或下一次正常關閉」才補推

commit 在 _save 內同步完成，硬殺最多丟「未 push」而非「未 commit」——資料都在本地 git 史裡，無資料遺失。自癒路徑存在：下次存檔 _commit 對乾淨樹回 True（:79-80 nothing to commit 分支）照樣排 push；即使整個 session 只看不改，正常關閉時 flush() 也無條件 _push。唯一缺口：硬殺後重開、又再被硬殺且期間無存檔 → 漏推跨多日，放大 F1 的分歧視窗。
